# Cascade Draft Ablation: Kernel × CUDA Graphs × Plan-Once

## TL;DR

We ablate three optimizations to the EAGLE draft phase (paged attention with CUDA graphs is SGLang's default; we replace the kernel and amortize the planner):

1. **Cascade attention kernel** — read shared prefix once across topk branches instead of `topk` times.
2. **CUDA graphs (CG)** — capture the multi-step draft as one CUDA graph and replay per iteration.
3. **Plan-once** — call the cascade scheduler once at max draft depth instead of N−1 times per iteration; per-step kv_len is patched on the GPU buffer via `update_draft_step`.

Headline result on Llama-3.1-8B + Llama-3.2-1B EAGLE draft (topk=10, depth=7, narrativeqa, 200 prompts, A6000 ×2, tp=2):

| step | cell | draft (s) | Δ |
|---|---|---|---|
| no optimizations (paged, no CG) | `paged_no_cg` | 4.297 | — |
| + CG (SGLang default) | `paged` | 4.068 | −0.229 (1.06×) |
| + cascade kernel (this work) | `cascade_per_step` | 2.416 | **−1.652 (1.68×)** |
| + plan-once (this work) | `cascade` | 2.194 | −0.222 (1.10×) |
| **Total** | | | **1.96× vs no-opt baseline, 1.85× vs SGLang default** |

The subtle finding: **the cascade kernel and CUDA graphs are super-additive, not independent**. The cascade kernel alone (without CG) yields essentially zero benefit; CG alone (with paged) yields 0.23 s; together they yield 1.88 s. The mechanism is a CPU dispatch ceiling that hides the cascade kernel's GPU speedup unless CG removes the per-call dispatch overhead. This is the key story of the ablation.

---

## Setup

**Models:** target Llama-3.1-8B, draft Llama-3.2-1B (STANDALONE EAGLE).

**Workload:** narrativeqa.jsonl, 200 prompts truncated to fit `context_length=110000`. `max_new_tokens=512`, batch size 1, greedy (`temperature=0`).

**Speculative decoding:** `eagle_topk=10`, `speculative_num_steps=7`, `speculative_num_draft_tokens=15`. Per draft iteration, the draft model produces a tree with 6 attention steps after the root (`speculative_num_steps - 1 = 6`).

**Hardware:** 2× NVIDIA RTX A6000 (48 GiB), `tp=2`. CUDA 12.8, mem fraction 0.40.

**Software:**
- `3rdparty/flashinfer` on branch `a6000_optimization` (provides `CascadeBatchAttentionWrapper.plan_for_draft` + `update_draft_step`).
- `3rdparty/sglang` fork with `DraftBackendFactory` env-var dispatch (env vars listed below).

**Per-prompt timing** comes from SGLang's internal `spec_draft_time` / `spec_verify_time` counters (enabled by `SGLANG_TIME_SPEC=1`). These add CUDA syncs but are isolated from end-to-end (e2e) wall clock.

**Output CSVs:** `results/A6000_standalone_<method>_Llama-3.1-8B_Llama-3.2-1B_top10_7_narrativeqa_temp0_bs1.csv` (one per cell), regenerated by `tests/bench_dataset.py`.

## Cells

We define the ablation space as a 2 × 3 grid (kernel × planning, with CG on/off). Plan-once is structurally cascade-only (flashinfer's flat wrapper does not expose shared planning), so two cells are not buildable.

| | per-step plan, no CG | per-step plan, CG | plan-once, no CG | plan-once, CG |
|---|---|---|---|---|
| paged kernel | `paged_no_cg` ✓ | `paged` ✓ | infeasible | infeasible |
| cascade kernel | `cascade_per_step_no_cg` ✓ | `cascade_per_step` ✓ | `cascade_no_cg` ✓ | `cascade` ✓ |

Six total cells, all benchmarked end-to-end on the same workload. Env vars and CSV names:

| cell | env var | CSV stem |
|---|---|---|
| `paged` | (none — SGLang default) | `standalone_paged_…` |
| `paged_no_cg` | `SGLANG_FLAT_DRAFT_NO_CG=1` | `standalone_paged_no_cg_…` |
| `cascade` | `SGLANG_CASCADE_DRAFT=1` | `standalone_cascade_…` |
| `cascade_no_cg` | `SGLANG_CASCADE_DRAFT_NO_CG=1` | `standalone_cascade_no_cg_…` |
| `cascade_per_step` | `SGLANG_CASCADE_PER_STEP_DRAFT=1` | `standalone_cascade_per_step_…` |
| `cascade_per_step_no_cg` | `SGLANG_CASCADE_PER_STEP_DRAFT_NO_CG=1` | `standalone_cascade_per_step_no_cg_…` |

The "no CG" variants set `supports_cuda_graph = False` on the multi-step backend. SGLang's `EAGLEWorker.init_cuda_graphs()` honors this flag and skips constructing the draft `EAGLEDraftCudaGraphRunner`. At inference time, `cuda_graph_runner is None`, so `eagle_worker.draft_forward()` takes the eager path that calls `init_forward_metadata` and dispatches each step's kernel individually. We verified this empirically with a one-shot debug print at `eagle_worker.py:584` showing `cuda_graph_runner is None: True` and `supports_cuda_graph=False`.

The `cascade_per_step` variants reuse the cascade attention kernel but plan it once per step (mirroring paged's per-step planning), instead of plan-once + `update_draft_step` patches.

| Method | Draft (s) | Decode (s) | Throughput (t/s) | AcceptLen |

| `ar` (no speculation) | 0.000 | 8.696 | 59.4 | — |
| `paged_no_cg` | 4.297 ± 0.810 | 6.280 ± 1.130 | 83.5 | 6.52 |

| `cascade_per_step_no_cg` | 4.312 ± 0.810 | 6.270 ± 1.130 | 83.6 | 6.51 |
| `cascade_per_step` | 2.416 ± 0.513 | 4.392 ± 0.891 | 120.2 | 6.47 |

`AcceptLen` is statistically identical across spec methods (6.47–6.52), confirming the kernel and planning changes preserve correctness — they only change wall-clock, not the draft tree's content.

## Per-optimization attribution

Shapley-style decomposition over the three feasible monotonic chains (kernel must precede plan-once, since plan-once is cascade-only):

| optimization | seconds saved (avg over orderings) | share of total |
|---|---|---|
| Cascade kernel | 0.541 | 26 % |
| CUDA graphs | 1.343 | **64 %** |
| Plan-once | 0.219 | 10 % |
| **Total** | **2.103** | (paged_no_cg → cascade) |

Marginal contributions per chain ordering:

| chain | + kernel | + CG | + plan-once |
|---|---|---|---|
| K → CG → PO | +0.015 (none) | **−1.896** | −0.222 |
| K → PO → CG | +0.015 (none) | −1.904 | −0.214 |
| CG → K → PO | −0.229 | **−1.652** | −0.222 |

The kernel's marginal benefit ranges from 0.015 s (when added before CG) to 1.652 s (when added after CG). This is the headline interaction effect.

## The cascade kernel and CUDA graphs are super-additive

Two paired comparisons make the interaction explicit:

| starting point | swap kernel | wall-clock benefit |
|---|---|---|
| `paged_no_cg` (4.30 s) | → `cascade_per_step_no_cg` (4.31 s) | **0.0 s** |
| `paged + CG` (4.07 s) | → `cascade_per_step + CG` (2.42 s) | **1.65 s** |

Same kernel swap, same workload — but the benefit jumps from 0 to 1.65 s depending on whether CG is enabled.

### Per-kernel-call analysis

The benchmark issues approximately 95k attention kernel calls (200 prompts × ~79 draft iterations × 6 steps per iteration = 94,800). Dividing each cell's draft time by this count:

| method | per-call time | bottleneck |
|---|---|---|
| `cascade + CG` | 23 µs | GPU compute |
| `cascade` no CG | 43 µs | **CPU dispatch** (GPU idle) |
| `paged + CG` | 43 µs | GPU compute |
| `paged` no CG | 45 µs | GPU compute (CPU concurrent) |

The Python + CUDA runtime cannot eagerly issue kernel launches faster than ~43 µs each (Triton + flashinfer JIT + cuLaunchKernel overhead, observed via `nsys`). This is a hard ceiling on per-call wall-clock in the no-CG regime.

- The **paged kernel's GPU compute time** per call is ~43 µs — already at the dispatch ceiling. Adding CG removes the ~2 µs of dispatch overhead that wasn't already concurrent with GPU work, hence only 1.06× speedup.
- The **cascade kernel's GPU compute time** is ~23 µs — well below the ceiling. Without CG, the GPU finishes in 23 µs and idles for 20 µs waiting for the next launch. The faster kernel is wasted because the CPU cannot feed it any faster. With CG, the entire multi-step draft replays as one graph; dispatch overhead is amortized to near zero, and the kernel runs at its true GPU throughput.

### Profiling evidence

`nsys` profiles of paged on a smaller workload (Llama-3.2-1B, 1.5k-token prompt, 64 output tokens, tp=1) confirm the dispatch story:

| CUDA API | `paged` (CG on) | `paged_no_cg` | ratio |
|---|---|---|---|
| Kernel launch APIs (cudaLaunchKernel + cuLaunchKernel + cudaLaunchKernelExC, summed) | 7 ms | **70 ms** | **10×** |
| `cudaDeviceSynchronize` | 210 ms | 25 ms | 8.4× lower |
| `cudaMemcpyAsync` | 34 ms | 35 ms | unchanged |
| `cudaGraphLaunch` | 2.6 ms | 1.9 ms | small |

Without CG, raw kernel launch APIs take 10× longer — direct evidence that the CG path is genuinely disabled and the eager dispatch overhead has appeared. With CG on, the process spends 82 % of CUDA-API time in `cudaDeviceSynchronize` (waiting on the GPU); without CG it spends only 18 % there and over 50 % issuing launches. The bottleneck literally moves from GPU to CPU.

On this small workload, paged is itself 2.25× slower without CG (0.197 s vs 0.443 s draft). The full bench's 1.06× is small because long-context per-step `plan()` does much more bookkeeping (longer kv_indices, longer cumsums, more in-place GPU→CPU syncs); that CPU work scales with prompt length while CG savings do not, so the CG benefit on paged decays from 2.25× to 1.06× as context grows from 1.5k to 55k tokens.

### Two different kernel families

Critically, **the paged and cascade backends do not use the same flashinfer kernel**. They are different kernel families with different schedulers, which we believe is the structural reason the dispatch ceiling pins them to similar wall-clock per call without CG.

| | paged draft | cascade |
|---|---|---|
| flashinfer kernel | `BatchPrefillWithPagedKVCacheKernel` (the prefill kernel, since draft q=topk per request) | persistent / "holistic" `BatchAttention` (`gen_batch_attention_module`, called `cascade_persistent_attention` in `csrc/batch_attention.cu`) |
| kernel calls per attention step | 1 main + 1 split-K merge (`PersistentVariableLengthMergeStatesKernel`) | 1 persistent main + 1 merge |
| where the scheduling cost lives | a thin per-launch grid setup in `plan()`; most work happens inside the kernel grid via standard split-K | a substantial C++ scheduler call inside `cascade_plan()` that pre-assigns work items to CTAs (the persistent kernel needs the assignment ahead of time) |
| benefit from CG | amortizes per-launch dispatch overhead (the kernel itself is unchanged) | amortizes per-launch dispatch overhead **and** the scheduler call (CG fast-replay path skips `cascade_plan()` entirely on replay) |

**Quantitative confirmation** (nsys count of `BatchPrefillWithPagedKVCacheKernel` launches on the small profile workload, 1 prompt, 64 output tokens):

- `paged + CG`: 32 explicit `cudaLaunchKernel` calls (just warmup + capture); production replay goes through `cudaGraphLaunch`.
- `paged_no_cg`: **896** explicit launches — every draft step's attention is dispatched eagerly.

The kernel binary is identical between these two; only the dispatch path changes. CG amortizes 28× of launches into a single graph replay, which is the source of paged's 1.06× CG benefit (the only thing CG buys on paged is dispatch amortization, since the kernel does not become faster).

For cascade, the same holds for the per-launch dispatch overhead, *plus* CG amortizes the more-expensive C++ scheduler call. This is why removing CG hurts cascade more than paged in absolute CPU-side cost. The cascade backend's `plan_once_for_all_steps` (used by `cascade_no_cg`) already mitigates this on the CPU side — collapsing N−1 plan calls to 1 — which is why `cascade_no_cg` (4.10 s) edges out `paged_no_cg` (4.30 s) by 5%, even though the kernel speedup itself is hidden by the CPU dispatch ceiling.

### Implication for the narrative

Treating the three optimizations as independent and additive is misleading. The honest framing is:

1. **Cascade kernel + CUDA graphs jointly contribute 1.88 s** (88 % of the total speedup). Neither alone delivers a meaningful win — they unlock each other. CG removes the CPU dispatch ceiling; the cascade kernel exploits the now-free GPU time.
2. **Plan-once contributes an additional, independent 0.22 s** (1.10× on top of kernel + CG) by reducing the cascade scheduler from N−1 calls to 1 per iteration.

A practical corollary: shipping the cascade kernel without recommending CUDA graphs would deliver almost no observable speedup. A user benchmarking it on workloads where CG is disabled (some exotic shapes, kernels that don't capture cleanly, debug builds) would conclude the kernel doesn't help. The cascade kernel and CG should be presented and adopted together.

## What about end-to-end?

Throughput improvements track draft-time improvements because verify time and accept length are essentially constant (Appendix A):

| Method | Throughput (t/s) | vs SGLang default |
|---|---|---|
| `ar` (no spec) | 59.4 | 0.67× |
| `paged` (SGLang) | 88.2 | 1.00× |
| `cascade_per_step_no_cg` | 83.6 | 0.95× |
| `cascade_no_cg` | 86.0 | 0.97× |
| `cascade_per_step` | 120.2 | 1.36× |
| `cascade` | 127.0 | **1.44×** |

End-to-end throughput speedup is smaller than draft-time speedup (1.44× vs 1.85× draft) because TTFT (prefill of the 55k-token prompt) is large and unaffected by draft-time optimizations. For shorter prompts where TTFT is negligible, the e2e and draft speedups would converge.

## Threats and caveats

- **Single workload.** Numbers are for one (model, draft, dataset, topk, depth, batch, hardware) point. The CPU dispatch ceiling story predicts that as context shrinks, paged's CG benefit grows (we verified this on the small profile workload: 2.25×). The cascade kernel's relative win should also grow at smaller context (less time spent in plan() bookkeeping that scales with prompt length). We have not measured the full grid yet.
- **AcceptLen drift.** `cascade_per_step` shows 6.47 vs 6.51 for the others — a 0.6% drift. This is within run-to-run variance for greedy generation across slightly different numerical orderings.
- **Plan-once cell for paged is infeasible.** Without modifying flashinfer's flat wrapper to support shared planning, we cannot ablate plan-once independently of the cascade kernel. The Shapley decomposition correctly handles this by averaging over only the three feasible orderings.
- **Run-to-run noise on a shared cluster.** TTFT differs by up to 0.6 s between paged and paged_no_cg runs even though prefill is independent of draft-time changes. This reflects shared-cluster noise, not the optimizations. Draft time is more stable because each prompt does ~80 draft iterations, averaging out per-launch jitter.
- **Two cells crashed once.** `cascade_per_step_no_cg` crashed twice with `cudaErrorIllegalAddress` when re-running at a short context (4k) on `a6000_optimization`. The full-bench (110k context) numbers reported here ran cleanly. This is a stability issue we have not yet root-caused; it does not affect the headline numbers.

## Reproducing

The orchestrating script is `tests/bench_dataset.py`. Each cell can be run with `--only`:

```bash
CUDA_VISIBLE_DEVICES=<i,j> CUDA_HOME=/usr/local/cuda-12.8 \
  uv run python tests/bench_dataset.py \
    --dataset-path data/narrativeqa.jsonl \
    --model-path meta-llama/Llama-3.1-8B \
    --draft-model-path meta-llama/Llama-3.2-1B \
    --eagle-topk 10 --speculative-num-steps 7 \
    --max-new-tokens 512 --context-length 110000 \
    --mem-fraction-static 0.40 --tp 2 \
    --time-spec --speculative-num-draft-tokens 15 \
    --result-prefix A6000_ \
    --only <paged|paged_no_cg|cascade|cascade_no_cg|cascade_per_step|cascade_per_step_no_cg>
```

The aggregator `scripts/aggregate_results.py` parses CSVs from `results/` and emits the comparison table including per-sample speedup standard deviations.

## Appendix A: verify time and accept length are constant

| Method | Verify (s) | AcceptLen |
|---|---|---|
| `paged_no_cg` | 1.579 | 6.52 |
| `paged` | 1.574 | 6.52 |
| `cascade_per_step_no_cg` | 1.555 | 6.51 |
| `cascade_per_step` | 1.572 | 6.47 |
| `cascade_no_cg` | 1.583 | 6.51 |
| `cascade` | 1.569 | 6.51 |

Verify time is dominated by the target model's verification of the proposed draft tree, which uses the standard target attention path and is unaffected by draft-side optimizations. AcceptLen is determined by the draft tree content and the target model's logits — both are unchanged by kernel/planning changes (modulo numerical noise). It is therefore safe to focus the ablation on draft time.
